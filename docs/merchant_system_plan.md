# 流浪商人系统计划

## 背景

当前已有补给箱、精英宝箱、守点事件和升级三选一，但还没有能让玩家主动规划路线和构筑的交易事件。商人系统的目标不是先做完整经济，而是增加一个短期地图目标：玩家看到商人后，需要判断是否冒险靠近、由谁交易、队友是否保护。

多人游戏中不能用全局暂停商店。一个玩家买东西时让全队暂停，会打断其他玩家正在处理的战斗、救援、精英和走位，体验很差。

## 第一版定位

第一版做成“流浪商人事件”，不是完整商店系统。

- 商人是限时地图事件。
- 商人出现后持续 30 到 45 秒。
- 每 3 到 4 分钟尝试出现一次。
- 同一时间最多存在 1 个商人。
- 玩家靠近商人交易圈后，可以打开自己的购买面板。
- 战斗不暂停，其他玩家完全不受影响。
- 商品是个人商品，不做共享库存。
- 交易后商人不立刻消失，其他玩家仍可交易。

## 多人交互规则

核心规则：实时交易，不全局暂停。

- 靠近商人的玩家才能打开商店。
- 打开商店时世界继续运行。
- 玩家仍然可以移动，自动武器继续攻击。
- 商店 UI 使用小面板，支持 `1/2/3` 快速购买，`Esc` 关闭。
- 玩家离开交易圈后，商店自动关闭。
- 玩家倒地后，商店自动关闭，并且不能购买。
- 商人到期离开后，所有打开的商店自动关闭。

商品库存建议按玩家独立生成：

- 每个玩家看到自己的 3 个商品。
- 一个玩家购买不会影响其他玩家的商品。
- 房主/服务器负责生成和验证商品，客户端只显示和发送购买请求。

## 受击打断规则

交易时被攻击不暂停、不无敌，直接打断交易。

- 玩家受到伤害时，商店面板立即关闭。
- 玩家正常掉血。
- 受击后 1.5 秒内不能重新打开商店。
- 商人不消失，玩家清怪后可以再回来交易。
- 如果玩家倒地，不能继续交易。
- 已经按下购买的交易立即结算，不回滚。
- 使用生命支付的商品必须保留至少 1 HP，不能直接把玩家扣死。

这个规则让商人变成有风险的目标，而不是免费安全点。

## 第一版商品

第一版只做 3 类商品，避免引入金币系统。

| 商品 | 效果 | 代价 | 设计目的 |
| --- | --- | --- | --- |
| 急救包 | 回复一段生命 | 消耗当前经验进度 | 救急，降低死亡挫败 |
| 武器调校 | 强化一件已有且未满级武器 | 扣除当前生命，最低保留 1 HP | 主动推进武器路线 |
| 禁忌遗物 | 获得一个随机遗物，优先优秀或稀有 | 降低本局最大生命 | 提供高风险高收益选择 |

推荐初始数值：

- 急救包：消耗 `xp_to_next * 35%` 的当前经验，回复 `max_hp * 35%`。
- 武器调校：扣除当前生命 `20%`，随机强化一件已有未满级武器。
- 禁忌遗物：最大生命 `-10`，获得一个未拥有遗物；当前生命同步压到新最大生命以内。

如果当前条件不满足，商品可以显示为不可购买：

- 当前经验不足，急救包不可购买。
- 没有可升级武器，武器调校不可购买。
- 遗物池为空，禁忌遗物不可购买。
- 当前生命不足以支付并保留 1 HP，生命支付商品不可购买。

## 暂不做

第一版明确不做这些内容：

- 金币掉落。
- 折扣。
- 刷新商品。
- 复杂库存。
- 共享商品抢购。
- 全员暂停。
- 投票暂停。
- 交易时无敌。
- 常驻基地商店。

这些内容等事件型商人验证好玩后再扩展。

## 实现方案

### 新增状态对象

新增 `scripts/entities/merchant_event_state.gd`。

建议字段：

- `position`
- `radius`
- `lifetime`
- `offer_seed`
- `opened_peer_ids`

商人本身只存位置、范围、剩余时间和随机种子，不直接处理购买逻辑。

### 扩展地图事件导演

扩展 `scripts/events/map_event_director.gd`：

- 增加商人计时器。
- 增加 `FIRST_MERCHANT_TIME`，建议 150 秒。
- 增加商人间隔，建议 180 到 240 秒。
- 增加 `MAX_ACTIVE_MERCHANTS = 1`。
- `update()` 返回 `merchant: true/false`。

### 主流程接入

扩展 `scripts/game/game.gd`：

- 新增 `merchant_events` 数组。
- `_update_map_events()` 中生成和更新商人。
- `_draw()` 中绘制商人和交易圈。
- 新增 `_update_merchant_events(delta)`。
- 新增 `_open_merchant_for_peer(peer_id, merchant)`。
- 新增 `_close_merchant_for_peer(peer_id)`。
- 新增 `_apply_merchant_purchase(peer_id, offer_index)`。

商店 UI 不应复用会暂停游戏的升级三选一流程。建议新增轻量商人面板，不设置 `choosing_upgrade = true`。

### 受击打断接入

需要在玩家实际受到伤害时通知商人系统。

推荐方式：

- 在主循环里调用 `take_damage()` 前记录玩家血量。
- 如果调用后血量下降，则执行 `_interrupt_merchant_for_peer(peer_id)`。
- `_interrupt_merchant_for_peer()` 关闭该玩家商店，并设置 `merchant_reopen_cooldowns[peer_id] = 1.5`。

这样不用先改 `Player` 成信号式结构，改动更集中。

### 网络同步

多人时由房主/服务器权威处理：

- 商人生成由房主决定。
- 商人位置、剩余时间通过世界快照同步。
- 每个玩家的商品列表由房主生成并同步给对应玩家。
- 客户端发送购买请求，只包含商人 ID 和商品索引。
- 房主验证玩家状态、距离、冷却、商品是否可买、代价是否足够。
- 验证通过后由房主扣代价并应用奖励。

需要避免客户端自己直接结算购买，否则容易出现不同步。

## UI 建议

商人面板要比升级三选一更轻：

- 面板靠屏幕右侧或下方，不遮挡中心战斗。
- 显示商人剩余时间。
- 显示 3 个商品、价格、效果。
- 支持 `1/2/3` 购买。
- 支持 `Esc` 关闭。
- 被攻击时显示短提示：`交易被打断`。
- 交易圈内显示明显边框和倒计时。

## 验收标准

- 多人中一个玩家打开商店时，其他玩家不暂停、不弹窗。
- 打开商店的玩家仍能移动，战斗继续运行。
- 交易玩家受伤后商店立刻关闭，并进入短暂重开冷却。
- 商品购买只影响购买者。
- 一个玩家购买不会移除其他玩家的商品。
- 商人到期后自动离开。
- 客户端不能在距离外、倒地、冷却中或代价不足时成功购买。
- 第一版不依赖金币，也能形成有意义的风险交易。

## 后续扩展

第一版稳定后再考虑：

- 金币掉落和正式经济。
- 商品刷新。
- 商人折扣。
- 特殊商人类型。
- 诅咒商品。
- 队伍共享的昂贵商品。
- 商人交易圈内轻微减速敌人，但不提供无敌。
